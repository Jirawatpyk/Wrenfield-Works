# i18n (TH/EN) Requirements Quality Checklist: Wrenfield Works Enterprise Site + CMS

**Purpose**: Validate that the bilingual (English/ไทย) requirements are complete, clear, consistent,
and measurable before implementation — a formal release-gate review of the spec's i18n quality.
**Created**: 2026-05-31
**Feature**: [spec.md](../spec.md)

**Note**: These items test the REQUIREMENTS, not the implementation. Each asks whether something is
properly specified, not whether it works. `[x]` = spec adequately covers it; `[ ]` = gap/ambiguity
(see Review Findings).

## Requirement Completeness

- [x] CHK001 Are the exact fields that require both EN and TH values enumerated, rather than left to interpretation? [Completeness, Spec §FR-013] — covered by data-model.md (per-field **L** flags)
- [x] CHK002 Is the default language for first-time visitors explicitly specified? [Completeness, Spec Assumptions]
- [x] CHK003 Are SEO/social-share metadata localization requirements (title, description, OG image) stated for both languages? [Completeness, Spec §FR-011a, §FR-015a]
- [x] CHK004 Are requirements defined for locale-aware formatting of numbers/currency (e.g., ฿ values shown in the design)? [Resolved → Spec §FR-011c: values authored per locale, no auto-reformat]
- [x] CHK005 Is there a requirement that every visible public string maps to a managed content key (none hard-coded)? [Completeness, Spec §FR-009]

## Requirement Clarity & Ambiguity

- [x] CHK006 Is the rule for which labels stay English ("mono"/technical) versus fully translated defined with objective, repeatable criteria? [Resolved → Spec §FR-011: explicit enumerated set]
- [x] CHK007 Can every string in the design be unambiguously classified as "narrative prose" or "technical label" using the spec's definition? [Resolved → Spec §FR-011 enumerated list + "everything else is prose"]
- [x] CHK008 Are the persistence rules for a visitor's language choice unambiguous (where stored, lifetime, behavior on invalid/stale value)? [Clarity, Spec §FR-003, Edge Cases] — invalid/stale handled; persistence implied by "return visits"
- [x] CHK009 Is the publish-completeness rule explicit about which states require both locales (draft may be incomplete, published may not)? [Clarity, Spec §FR-014]

## Requirement Consistency

- [x] CHK010 Is the SEO-metadata bilingual requirement consistent with the general publish-completeness gate (both treated the same way)? [Consistency, Spec §FR-011a vs §FR-014]
- [x] CHK011 Is the language-switch latency target stated once and consistently (no conflicting numbers)? [Consistency, Spec §SC-001]
- [x] CHK012 Are brand/client names (logos) consistently specified as non-translated across spec and data model? [Consistency, data-model §ClientLogos]

## Measurability / Acceptance Criteria

- [x] CHK013 Are Thai typography requirements (line-height, display size, no negative tracking, no clipping) quantified or only described qualitatively? [Resolved → Spec §FR-004: line-height ≥1.25, no negative tracking, reduced Thai scale, no clipping]
- [x] CHK014 Can "all narrative prose updates on switch" be objectively verified (is completeness measurable)? [Measurability, Spec §FR-002, §SC-003]
- [x] CHK015 Is SC-001 (switch < 1s, preference remembered) expressed so it can be tested without implementation knowledge? [Measurability, Spec §SC-001]

## Coverage & Edge Cases

- [x] CHK016 Is graceful fallback when only one locale is populated defined as a binding requirement, not merely noted? [Coverage, Edge Cases]
- [x] CHK017 Does the spec define behavior when a collection item is added/reordered in one locale but not the other? [Resolved → Spec Edge Cases "Per-locale item existence/order": existence/order shared, only values localized]
- [x] CHK018 Is the per-locale routing/URL strategy addressed at requirement level or explicitly deferred to planning? [Gap] — addressed in plan.md (locale segments / proxy.ts)
- [x] CHK019 Are requirements defined for what a returning visitor sees if their saved language is no longer valid? [Coverage, Edge Cases]

## Review Findings — 2026-05-31 (RESOLVED: 19/19 pass)

All five original gaps were resolved by spec edits on 2026-05-31:

- **CHK004** → Spec §FR-011c (numeric/currency authored per locale; no auto-reformat).
- **CHK006 / CHK007** → Spec §FR-011 rewritten with an explicit enumerated English-only set;
  everything not in the set is translatable prose.
- **CHK013** → Spec §FR-004 now quantified (line-height ≥ 1.25, no negative tracking, reduced Thai
  display scale, no clipping/overlap).
- **CHK017** → Spec Edge Cases "Per-locale item existence/order" (existence/order shared across
  locales; only field values localized).
